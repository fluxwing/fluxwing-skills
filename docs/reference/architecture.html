Architecture Deep Dive

Peek inside Fluxwing’s plugin design, learn how the uxscii standard keeps outputs portable, and see how the file system, agents, and validation layers connect.

Design Principles

Fluxwing is built for AI-native collaboration. Every decision keeps ASCII-first workflows fast, portable, and human-friendly.

  • AI-native format — ASCII layouts + JSON metadata are easy for models to read and write.
  • Fluxwing vs uxscii — Fluxwing is the agent; uxscii is the open standard it speaks.
  • Self-contained bundle — schemas, examples, and docs ship with the plugin.
  • Rendered truth — every screen includes a real-data `.rendered.md` example.
  • Modular docs — small, targeted files keep agent context lean.

Plugin Layout

The plugin directory groups commands, agents, data, and documentation. Everything required for uxscii design ships together. 

plugin/
├─ commands/          # Slash command manifests
├─ agents/            # Autonomous agent manifests
├─ data/
│  ├─ schema/         # JSON Schema validation
│  ├─ examples/       # Component templates (11x)
│  ├─ screens/        # Screen examples (2x)
│  └─ docs/           # Modular documentation blocks
└─ README.md, COMMANDS.md, AGENTS.md, ARCHITECTURE.md …

Users never edit bundled files directly. Instead, Fluxwing writes to the project workspace located at ./fluxwing/.

Workspace Outputs

Fluxwing splits outputs into components, screens, and library overrides so you can track provenance. 

./fluxwing/
├─ components/        # {name}.uxm + {name}.md
├─ screens/           # {screen}.uxm/.md/.rendered.md
└─ library/           # Customized template copies
Remember: read from bundled references, write to the project workspace. Upgrades stay painless, and your tailor-made designs remain untouched.

The uxscii Standard

uxscii pairs structured JSON (.uxm) with readable ASCII templates (.md). Together, they capture intent, metadata, and visual layout in formats that humans and AI can improve collaboratively.

Two-File Contract

  • Metadata — IDs, versions, props, accessibility affordances, relationships.
  • ASCII template — Layout, spacing, and states using human-readable characters.

Screens extend the contract with a third .rendered.md file that demonstrates real data in context.

Component Hierarchy

Fluxwing follows an atomic → composite → screen hierarchy:

  1. Atomic — buttons, inputs, icons, text elements.
  2. Composite — cards, forms, navigation built from atomic pieces.
  3. Screens — full experiences referencing both atomic and composite components.

Agents and commands respect this order when building new designs, ensuring dependencies are ready before they are referenced.

Validation & Quality

A single JSON Schema powers the validator. Both commands and agents rely on it to guarantee consistency across the ecosystem.

  • Schema checks for required metadata, props, states, and relationships.
  • Accessibility rules enforce roles, labels, and interaction hints.
  • Rendered examples keep ASCII intent aligned with real-world data.

Combining schema validation with manual review in ASCII shortens the feedback loop between designers, engineers, and AI agents.

Extensibility

Ready to go beyond the defaults? Fluxwing was designed for safe extension.

  • Additional commands — drop new manifests under commands/.
  • Custom agents — use the patterns demonstrated by the designer, validator, and composer.
  • Helpers & hooks — reserved directories support future integrations without breaking structure.
  • Library management — keep reusable variants in ./fluxwing/library/ for teams.
