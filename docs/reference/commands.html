Fluxwing Documentation

Slash Command Reference

Learn how each Fluxwing command accelerates ASCII-first design inside Claude Code. Understand when to use them, what they create, and how the generated files slot into your project workspace.

At a Glance

Fluxwing ships with six high-impact commands. Use them to create components, scaffold entire screens, validate quality, explore the bundled library, inspect existing assets, and import UI screenshots—all without leaving the terminal.

Command Primary Goal Outputs
/fluxwing-create Create a single component with metadata + ASCII template components/{name}.uxm, components/{name}.md
/fluxwing-scaffold Generate a complete screen and any missing components screens/{name}.uxm, .md, .rendered.md
/fluxwing-validate Audit components or screens for schema + accessibility issues Terminal quality report, optional fixes
/fluxwing-library Browse bundled templates and project components Interactive terminal menu
/fluxwing-get Inspect a specific component with context On-demand ASCII preview + metadata summary
/fluxwing-import-screenshot Convert UI screenshots into uxscii components automatically Multiple components + screen files from visual analysis
Tip: Commands always write to your local ./fluxwing workspace. Bundled examples remain read-only, so you can experiment without risking base templates.

Data Locations

Fluxwing strictly separates bundled resources from your editable project files. Commands rely on that separation to keep upgrades predictable and your work safe.

Read-Only Reference

  • {PLUGIN_ROOT}/data/examples/ — 11 curated component templates
  • {PLUGIN_ROOT}/data/screens/ — 2 rendered screen examples
  • {PLUGIN_ROOT}/data/docs/ — Documentation blocks for agents
  • {PLUGIN_ROOT}/data/schema/ — JSON Schema definitions

Your Writable Workspace

  • ./fluxwing/components/ — Components you create or customize
  • ./fluxwing/screens/ — Screen compositions, including rendered examples
  • ./fluxwing/library/ — Template copies you tailor for reuse

Whenever Fluxwing needs to check what already exists, it searches in priority order: components → library → bundled examples. That keeps your bespoke designs front and center.

/fluxwing-create: Component Builder

The fastest way to craft structured components. Fluxwing guides you through naming, states, props, and templating, then delivers ASCII and metadata files ready for iteration. 

/fluxwing-create pricing-card

Workflow

  1. Plan name, component type, props, and required states.
  2. Optionally reuse a bundled template or start from scratch.
  3. Fluxwing generates a matching .uxm file and ASCII template.
  4. Outputs are validated and saved to ./fluxwing/components/.

Best Practices

  • Choose descriptive, kebab-case names such as primary-cta-button.
  • Document all interaction states (default, hover, focus, disabled).
  • Reference bundled examples when you need quick inspiration.

/fluxwing-scaffold: Screen Composer

Build full screens with one command. Fluxwing inventories required components, creates missing pieces, and assembles everything into a complete screen package. 

/fluxwing-scaffold onboarding-wizard

What You Get

  • {screen}.uxm — Screen metadata with component references.
  • {screen}.md — Templated ASCII blueprint with {{variables}}.
  • {screen}.rendered.md — Example output filled with representative data.

Any missing components are created automatically, validated, and stored alongside your existing library. Fluxwing follows an atomic → composite → screen build order to keep dependencies tidy.

/fluxwing-validate: Quality Reports

Run validation whenever you need confidence in a component or screen. Fluxwing checks schema compliance, accessibility attributes, and metadata completeness before highlighting next steps. 

/fluxwing-validate screens/login-screen.uxm

Pair validations with the designer or composer agents to keep large projects harmonized as they grow.

Discovery & Inspection Commands

/fluxwing-library

Browse every available component — bundled templates, customized library entries, and newly created assets — using an interactive terminal menu. Perfect for onboarding teammates or reviewing available building blocks.

/fluxwing-get

Display the ASCII preview, metadata, and usage tips for a specific component without leaving your current context. Ideal for quick refreshers during implementation.

/fluxwing-import-screenshot: Screenshot to uxscii

Transform UI screenshots into production-ready uxscii components automatically. Fluxwing analyzes the visual layout, extracts components, generates ASCII representations, and creates complete metadata files—all from a single image. 

/fluxwing-import-screenshot path/to/screenshot.png

What You Get

  • Atomic Components — Buttons, inputs, badges, icons extracted as individual components
  • Composite Components — Forms, cards, navigation assembled from atomics
  • Complete Screen — Full layout with .uxm, .md, and .rendered.md files
  • Validation Report — Schema compliance and quality metrics for all generated files

How It Works

  1. Visual Analysis — AI-powered computer vision detects all UI components and layout structure
  2. Component Extraction — Each element is classified (button, input, card, etc.) with visual properties
  3. ASCII Generation — Creates faithful ASCII representations matching the original design
  4. Metadata Creation — Generates complete .uxm files with props, states, and accessibility
  5. Validation — Ensures all generated files meet uxscii standards

Supported Formats

PNG, JPG, JPEG, WebP, GIF

Best Practices

  • Use clear, high-resolution screenshots for better detection accuracy
  • Break complex interfaces into focused sections (one feature at a time)
  • Review and customize generated components to match your brand
  • Run /fluxwing-validate after import to check quality
Tip: Screenshot import works best with standard UI patterns. Custom or highly stylized interfaces may require manual adjustments after generation.
